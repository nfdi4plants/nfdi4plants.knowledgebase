## ARC revision steps

<Steps>

1. Once an ARC is submitted as RFP, all ARC reviewers receive an email
2. If you plan to review an ARC, please communicate this by responding directly to the reviewer email list
3. Open the RFP on the ARChive https://archive.nfdi4plants.org

      :::note[WIP July 2026]
      Currently, all ARC reviewers use the same invenio account
      :::

4. Respond to the RFP, e.g. that "a DataPLANT reviewer" (closed review) or you (open review; e.g. add name and affiliation) will initiate the revision. In the response, provide a link to the ARC issue board.

    :::note
    This response will be visible to the ARC submitter via the [ARChigators publication status](/nfdi4plants.knowledgebase/arc-publications/arc-publication-02-request/#publication-status)
    :::

5. Initiate the ARC revision
    - Inspect the ARC in the DataHUB
    - Download and inspect the ARC locally
    - Add reviewer comments as work items / issues directly to the ARC in the DataHUB
    - Consider using the default issues provided below


</Steps>

:::note[WIP July 2026]
- A mechanism to assign ARC reviewer DataHUB accounts to RFP-ARCs is currently work in progress.
- Oftentimes ARC reviewers already have access to the ARC under revision due to their data steward role in the project.
- If not, this could be requested by contacting the ARC submitters via invenio / the publication status page
:::


## Using default issues for ARC revision

This guide provides a template issue list for ARC revision in the DataHUB. The issue list can be imported into the DataHUB to create a default set of issues for ARC revision. The issue list is based on the typical review criteria for ARC publication in DataPLANT, but can be adapted to the specific needs of the ARC revision.

:::note
In the long run, the revision will be automated as much as possible via the [arc-validation](/nfdi4plants.knowledgebase/arc-validation/) workflows and validation packages, but for now, this issue list can be imported manually into the DataHUB. This issue-based revision may also help to engage with the ARC authors and reviewers in the DataHUB to transparently discuss the revision process and the quality criteria for ARC publication.
:::

## Using the review issue list during ARC revision

<Steps>

1. Navigate to the ARC to be revised in the DataHUB
2. Navigate to labels: left sidebar → Manage → Labels and click `Generate a default set of labels`
    :::note
    This enables to tag issues with default GitLab labels such as `critical`, `enhancement`, `suggestion` as provided in the following issue list.
    :::
1. Open the issue menu: left sidebar → Plan → Work items
1. Via the three-dots menu in the top-right corner use `Import CSV`
1. Select the `review-arc-issues.csv` file and import it
1. Wait a short moment. You receive an email notification once the import finished (or failed due to parsing errors).

</Steps>

## (Optional) Bundle issues in a milestone

<Steps>

1. Create a new milestone (left sidebar → Plan → Milestones), e.g. with the following title and description:

    **Title**
    
        `ARC publication revision`

    **Description**
    
        ```markdown showLineNumbers=false nowrap=true

        This milestone helps to bundle all issues (➡ [Issues](../../issues)) related to the ongoing revision of your ARC for publication via DataPLANT's [ARChive](https://archive.nfdi4plants.org/).

        ### ARC publication criteria

        [This article](https://nfdi4plants.github.io/nfdi4plants.knowledgebase/guides/review-arc/) in the knowledge base summarizes what DataPLANT typically expects during ARC publication.

        The ➡ [Issues](../../issues) are used to transparently discuss certain quality criteria for the ARCs. Feel free to get in touch with the reviewers and data stewards by responding directly via the issues.

        ### Issue names and labels

        - Issues starting with `DataHUB:` can easily be addressed here in the DataHUB
        - Issues starting with `ISA Metadata:` are most easily fixed using some ARC tooling, e.g. ARCitect, Swate, ARC commander
        - Issues labeled as `critical` **must** be addressed for acceptance
        - Issues labeled as `suggestion` **should** be addressed for acceptance – this also depends on the nature of the ARC
        - Issues labeled as `enhancement` add a nice touch to the ARC, e.g. for (human) findability
        ```

1. Open the work items menu: left sidebar → Plan → Work items
1. Click on the "Bulk edit" button in the top-right corner to select some or all issues
1. In the right sidebar, select the "Milestone" dropdown and choose the milestone you just created
1. Click on "Update selected" to apply the milestone to the selected issues

</Steps>

:::tip[Close multiple issues via bulk edit]
In case there's an issue with the import or you want to immediately close (some of) the imported issues, you can also use the `bulk edit` feature. Select the relevant issues, change the status to "Close", select "Update selected".
:::


## What's next? – The actual revision process

The issue list is really just a starting point for ARC revision. This should help ARC reviewers to have a clear overview of the most important issues that need to be addressed for ARC publication. However, the issue list is not exhaustive and may need to be adapted to the specific needs of the ARC revision. For example, some issues may not be relevant for a specific ARC, while other issues may need to be added. 

In order to guide the ARC creators and relate the issues to their specific ARC, it usually makes sense to add some comments to the issues. This can be done directly in the DataHUB by responding to the issues and include specific parts, e.g. study or assay names and screenshots (e.g. from ARCitect).

Furthermore, since the issue list is generic and with the logic above, always all issues are imported, it may be useful to close some of the issues immediately after import if they are not (anymore) relevant for the specific ARC revision. Again, this highlights, that certain criteria simply apply to all ARCs.

:::tip[A pull-request goes a long way]
For small things such as adding a License or the automated README, ARC creators usually also appreciate a pull request. 
:::


## Adapting the issue list

The issue list is based on the [CSV import feature of GitLab issues](https://docs.gitlab.com/user/project/issues/csv_import/), which allows you to create multiple issues at once by uploading a CSV file with the issue details. For more details on the CSV format and the available options, please refer to the GitLab documentation. 

### Basics about the issue CSV file

- The first two columns must be `title` and `description`.
- The first row is used as a header and is ignored. 
- The order of the rows does not matter – by default the issues are sorted alphabetically in the DataHUB.

### Use of quick-actions

GitLab offers so-called "quick-actions" to perform certain actions on issues, e.g. to assign an issue to a user, to add a label or to mark an issue as confidential. These quick-actions can also be used in the CSV file to set up the issues in a more specific way during import. 

- Each action must be on a separate line.
- For quick actions like `/label` and `/milestone`, the label or milestone must already exist in the project.

#### Labels

Using the quick-action `/label` allows to categorize the issues. Instead of typical journal review terminology (e.g. "required", "recommended", "major", "minor"), this issue list uses the default GitLab labels `critical`, `suggestion` and `enhancement`. 

:::tip
You can see all available labels via left sidebar → Manage → Labels
:::

#### Confidential

Marking review issues as "confidential" via the `/confidential` quick-action simply adds a bit more comfort for the ARC authors. Issues marked as confidential are only visible to project members with at least "Reporter" permissions.
